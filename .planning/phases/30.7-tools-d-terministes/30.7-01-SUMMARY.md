## Decisions Made

- **PEP 420 namespace package for tools/** — repo-root `tools/checks/` is a namespace (no `__init__.py`). Creating `tools/mcp/mint-tools/tools/__init__.py` (as the plan template suggested) makes `tools` a regular package, which pytest resolves to the nearest match first and breaks Wave 0 tests importing `tools.checks.accent_lint_fr`. Fix: delete the `__init__.py`; both locations now co-exist as namespace-package portions.
- **tax → capital_tax prefix mapping** — CONTEXT.md Area 2 lists `tax` as a canonical category, but the registry's actual key prefix is `capital_tax.*`. The public contract keeps the short name; the internal map translates once at the call boundary. 33 entries returned for `tax` (mainly cantonal capital_tax rates).
- **Scalar-only filter** — `avs.echelle44` has a 26-row list as its `.value` (a regulatory table). `ConstantEntry.value: float` cannot hold it; the scalar-only filter skips it cleanly rather than crashing. Exposing tabular entries is explicitly CONTEXT.md deferred scope.
- **app.models namespace shim** — `app/models/__init__.py` eagerly imports sqlalchemy-backed ORM classes. The MCP venv has neither sqlalchemy nor DB drivers. Pre-registering `sys.modules['app.models']` as an empty namespace package before the registry imports its leaf dataclass skips `__init__.py` execution. Same pattern reused in both tool modules.
- **Module-scope lazy singleton for ComplianceGuard** — `_GUARD: Any = None` + `_get_guard()` instantiates on first real call rather than at module import. Keeps startup fast if the MCP server loads both tool modules but only ever routes to `constants.py`. Test `test_module_scope_guard_singleton` asserts identity across calls.
- **Single-line `from app.services.coach.compliance_guard import ComplianceGuard`** — adopted to satisfy the plan's literal grep acceptance check; continuation-line form would have passed the test suite but not the grep.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 — Bug] Plan's `parents[3]` off-by-one in constants.py PYTHONPATH injection**
- **Found during:** Task 1 GREEN (tests failed with empty `ConstantsResult.constants` lists)
- **Issue:** Plan template computed `_REPO_ROOT = Path(__file__).resolve().parents[3]`. `constants.py` lives at `tools/mcp/mint-tools/tools/constants.py`, which is 5 levels under the repo root: `parents[0]=tools`, `[1]=mint-tools`, `[2]=mcp`, `[3]=tools`, `[4]=repo root`. With `[3]`, the injected path pointed to `repo/tools/services/backend` (non-existent) and the backend import silently failed via the `try/except ImportError` fallback.
- **Fix:** Changed to `parents[4]`. This is the exact same off-by-one documented in 30.7-00-SUMMARY.md §Deviations item 1 — flagged as a recurring plan-template error.
- **Files modified:** `tools/mcp/mint-tools/tools/constants.py`, `tools/mcp/mint-tools/tools/banned_terms.py` (same fix applied prophylactically).
- **Committed in:** `6a99bc87` (TOOL-01), `9f4eaf22` (TOOL-02).

**2. [Rule 3 — Blocking] `app.models.__init__` pulls in sqlalchemy; MCP venv has no ORM deps**
- **Found during:** Task 1 GREEN (direct backend import failed with `ModuleNotFoundError: No module named 'sqlalchemy'`)
- **Issue:** `services/backend/app/models/__init__.py` eagerly imports `User`, `ProfileModel`, `SessionModel`, 20+ other ORM classes. Importing `app.services.regulatory.registry` (which imports `app.models.regulatory_parameter`) triggers `app.models.__init__` to execute. The MCP venv intentionally has no sqlalchemy, bcrypt, cryptography, etc.
- **Fix:** Added `_ensure_lightweight_app_namespace()` helper in both tool modules. Pre-registers `app` and `app.models` as empty namespace packages in `sys.modules` before the registry/guard imports run. Python's import machinery then skips `app/models/__init__.py` and loads `app.models.regulatory_parameter` (a pure dataclass, no DB deps) directly.
- **Alternative considered:** Install sqlalchemy + DB deps into MCP venv. Rejected: bloats venv, couples MCP dev-time tooling to backend runtime ORM, violates the minimal-install goal from CONTEXT.md Area 1.
- **Files modified:** same two tool files as above.
- **Committed in:** `6a99bc87`, `9f4eaf22`.

**3. [Rule 1 — Bug] Empty `tools/__init__.py` shadowed repo-root `tools.checks.*` namespace**
- **Found during:** Task 1 GREEN, right after app.models shim fixed the import path (full suite regressed from 21 passed → 1 collection error)
- **Issue:** The plan's Task 1 step 1 said "Create `tools/mcp/mint-tools/tools/__init__.py` — empty file. Python package marker." Once created, `tools` became a regular package rooted at `tools/mcp/mint-tools/tools/`. Repo-root `tools/checks/` is a namespace package (no `__init__.py`). Python's module resolver found the regular package first and Wave 0's `test_accent_lint_scan_text.py` (which imports `from tools.checks.accent_lint_fr import ...`) started failing with `ModuleNotFoundError: No module named 'tools.checks'`.
- **Fix:** Deleted `tools/mcp/mint-tools/tools/__init__.py`. `tools` is now a PEP 420 namespace package at every depth. Both `tools.constants`, `tools.banned_terms`, and `tools.checks.accent_lint_fr` resolve under the same namespace, and the full suite is green.
- **Files modified:** `tools/mcp/mint-tools/tools/__init__.py` deleted.
- **Committed in:** `6a99bc87` (deletion bundled with the GREEN commit).

**4. [Rule 1 — Bug] Tabular registry entry (`avs.echelle44`) crashed `float()` coercion**
- **Found during:** Task 1 GREEN (avs parametrized test failed with `TypeError: float() argument must be a string or a real number, not 'list'`)
- **Issue:** `RegulatoryParameter.value: float` is the declared type but `avs.echelle44` stores a 26-row table as a Python `list` (documented in CONTEXT.md §deferred as "26-row table not currently in RegulatoryRegistry" — actually it IS in the registry as a list, just not reachable by the tool contract).
- **Fix:** Added scalar-only filter — entries whose `.value` is not `(int, float, bool)` are skipped. avs now returns 27 scalar entries instead of crashing on the 28th.
- **Files modified:** `tools/mcp/mint-tools/tools/constants.py`.
- **Committed in:** `6a99bc87`.

**5. [Rule 1 — Bug] Plan's interface hint assumed `_check_banned_terms` returns list[dict]; reality is list[str]**
- **Found during:** Task 2 GREEN (verified upstream signature before writing the wrapper)
- **Issue:** Plan's `<interfaces>` block said `def _check_banned_terms(self, text: str) -> list: ...  # returns list[dict] with keys: "term", "context", "position"`. Actual signature returns `list[str]` (bare term labels). The tool's normalisation code handled both shapes, but the test expectation had to match reality.
- **Fix:** Wrote the tool to normalise both list[str] and list[dict] inputs for forward compatibility. Tests assert against the str-path (the real one). Context window is built from the scanned text rather than expecting it in the upstream payload.
- **Files modified:** `tools/mcp/mint-tools/tools/banned_terms.py`.
- **Committed in:** `9f4eaf22`.
