## Decisions Made

- **Fallback envelope shape matches success envelope** — Same Pydantic model `ArbParityResult` in all five status branches. Agents pattern-match on `.status`, never on presence/absence of fields. `exit_code=None` in the fallback branch is still a typed field.
- **Tail truncation, not head** — Long drift reports put the actionable errors at the BOTTOM of stdout/stderr. The `_truncate` helper preserves the tail and prepends a `"...[truncated]..."` marker so the caller can see that data was dropped.
- **`cwd=_REPO_ROOT` on the subprocess** — `arb_parity.py` (when Phase 34 ships it) will read ARB files by relative paths. Setting `cwd` explicitly avoids a later ambiguity where the tool would otherwise inherit the MCP server's spawn directory.
- **Narrowed sys.executable regression test during Task 1 GREEN** — The first version of `test_module_uses_sys_executable_not_literal_python3` grepped the whole file for `"python3"`, which caught the docstring's legitimate mention of "host default python3 is 3.9.6". Fixed the test to skip comment / docstring lines and check for subprocess-call-argv patterns only (`["python3"` / `['python3'` / `["python3.11"` / `['python3.11'`). Tool module itself unchanged.
- **`MAX_TEXT_LEN = 10_000` mirrors TOOL-02** — Consistent DoS cap policy across all text-input tools in this phase. If future requirements demand different caps, change in one place per tool, not across a rule set.
- **Scan the cap-truncated prefix but report original length** — `text_length` returns `len(text)` BEFORE the `[:MAX_TEXT_LEN]` slice so downstream callers (Wave 2, agents) can detect and warn about truncation. Silent capping would create a gap between the visible response and the actual scan surface.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 1 — Bug] Over-strict sys.executable regression test caught the module's docstring**
- **Found during:** Task 1 GREEN (7/8 passed; the subprocess-regression test failed)
- **Issue:** The plan sketch for the regression test used `assert '"python3"' not in src` on the full file contents. The module's docstring explains the Pitfall 2 design decision with a line like `"host default python3 is 3.9.6"`, which contains the literal substring `"python3"`. The assertion fired on documentation, not on code.
- **Fix:** Rewrote the test to iterate over `src.splitlines()`, skip lines starting with `#`, `"`, or `'` (comments and docstrings), and check only the remaining code lines for forbidden argv-opening patterns (`["python3"`, `['python3'`, `["python3.11"`, `['python3.11'`). The original intent — "no literal python3 in the subprocess call" — is preserved and strengthened (also catches the 3.11 variant, which is load-bearing for future readers).
- **Files modified:** `tools/mcp/mint-tools/tests/test_validate_arb_parity.py` only; the tool module was correct from the start.
- **Verification:** `.venv/bin/pytest tests/test_validate_arb_parity.py -v` → 8/8 passed after the fix.
- **Committed in:** `49a7197f` (bundled into Task 1 GREEN — the test and the module landed in the same commit).
