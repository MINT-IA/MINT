<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client -> /anonymous/chat | Untrusted anonymous input crosses into LLM pipeline |
| X-Anonymous-Session header | Client-supplied device token used for rate limiting |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-13-01 | Tampering | X-Anonymous-Session header | mitigate | Validate UUID format (regex `^[a-f0-9-]{36}$`), reject malformed tokens. IP fallback rate limit as secondary defense. |
| T-13-02 | Tampering | message field | mitigate | Pydantic validator (max 2000 chars, strip whitespace), PII scrubbing via `_scrub_pii()`, prompt injection defense via existing `_INJECTION_PATTERNS` |
| T-13-03 | Denial of Service | /anonymous/chat endpoint | mitigate | 3-message lifetime limit per device token + IP-based slowapi rate limit as secondary (e.g., 10/minute per IP) |
| T-13-04 | Spoofing | UUID farming (generate many UUIDs) | accept | Acceptable risk: each UUID gets only 3 messages (low value), IP fallback provides secondary defense, no PII at stake |
| T-13-05 | Information Disclosure | LLM response leaks internal capabilities | mitigate | Discovery prompt written from scratch (not derived from authenticated prompt), no tools/profile/memory references, ComplianceGuardrails applied |
| T-13-06 | Elevation of Privilege | Anonymous endpoint accessing authenticated resources | mitigate | Separate router, no `get_current_user` dependency, no profile/memory/RAG access, separate DB model |
</threat_model>
