# Plan 53-02 — `SequenceChatHandler` wiring end-to-end (`retirementPrep` template) (PLAN)

**Status:** Ready (depends on Plan 53-01 PR-2 (#445) + walker scripts (#443) merged)
**Companion CONTEXT:** `53-CONTEXT.md`
**Estimated effort:** 2–3 days

## Goal (one sentence)

Activate the `retirementPrep` SequenceTemplate end-to-end so that after a `retirement_choice` (or `retirement_projection`) intent is detected in coach chat, the user is routed to the appropriate screen via `RoutePlanner`, and on return the chat picks up at the next sequence step via `SequenceChatHandler.handleStepReturn` — replacing today's « screen return = static contextLine » dead-end with a real multi-screen journey.

## Why this plan now

Expert 5 (Engineering / Wiring Reviewer) called out:
- 10 declared `SequenceTemplate`s in `apps/mobile/lib/models/sequence_template.dart:88-577` with **ZERO production callers** (only test files import `SequenceChatHandler`).
- The 1411-line `coach_chat_screen.dart` never imports `SequenceChatHandler`.
- The walker logs show « 4 identical 167 076-byte screenshots » — proof that the canvas was never reached because there's no return loop.

Plan 53-01 closed the parity foundation. Plan 53-02 lights up the sequence pipe with ONE template end-to-end. The other 9 templates stay on the shelf for Phase 55+.

## Current wiring (read 2026-05-04)

- `coach_chat_screen.dart:198` declares `_screenReturnSub` listening on `ScreenCompletionTracker.stream`.
- `coach_chat_screen.dart:419 _subscribeToScreenReturns` builds a `contextLine` string from the screen return and stores it in `_entryPayloadContext` for the next chat message.
- `SequenceChatHandler.handleStepReturn` (`sequence_chat_handler.dart:63`) IS implemented but never called.
- `SequenceChatHandler.startSequence` (`:187`) IS implemented but never called.
- `SequenceTemplate.retirementPrep` is keyed by intent at `sequence_template.dart:581` (`'retirement_choice' || 'retirement_projection' => retirementPrep`).
- `SequenceCoordinator` (`sequence_coordinator.dart:82`) handles `AdvanceAction` / `CompleteAction` / `PauseAction` / `SkipAction` / `RetryAction` / `ReEvaluateAction`.

The integration point is `_subscribeToScreenReturns`: today it builds a contextLine; with this plan it ALSO consults `SequenceChatHandler.isSequenceActive()` and dispatches to `handleStepReturn` when active.

### T-53-02-02 — Wire `handleStepReturn` into `_subscribeToScreenReturns`

**Files:**
- `apps/mobile/lib/screens/coach/coach_chat_screen.dart:419-432` — extend `_subscribeToScreenReturns`. New flow:

```dart
void _subscribeToScreenReturns() {
  _screenReturnSub = ScreenCompletionTracker.stream.listen((screenReturn) async {
    if (!mounted) return;

    // Phase 53-02 — if a sequence is active, dispatch through the handler
    // first so the next prompt is sequence-driven, not just contextLine-driven.
    if (await SequenceChatHandler.isSequenceActive()) {
      final result = await SequenceChatHandler.handleStepReturn(
        outcome: _outcomeFromScreenReturn(screenReturn),
        screenReturn: screenReturn,
        // ... other params per existing signature
      );
      if (result != null) {
        _injectSequencePrompt(result);
        return; // sequence handler owns the next prompt; skip contextLine path
      }
    }

    // Existing fallback: build contextLine for the next user message.
    final fields = screenReturn.updatedFields;
    final fieldSummary = ...;
    _entryPayloadContext = ...;
  });
}
```

Plus a new helper `_injectSequencePrompt(SequenceHandlerResult result)` that pushes the next prompt as a coach-side message (same path as the existing silent-opener / proactive prompt rendering).

**Behavior:**
- When a sequence is active and a screen return fires, the sequence handler's next-step prompt is pushed into the chat as a coach message.
- When no sequence is active, the existing contextLine flow runs unchanged.
- The dispatcher gates on `SequenceChatHandler.isSequenceActive()` so non-sequence flows are unaffected.

**Test:** widget test mocks `SequenceChatHandler.isSequenceActive` returning true + `handleStepReturn` returning a fake `SequenceHandlerResult`; asserts the next coach message contains the result's prompt text.

## Risks + mitigations

| Risk | Mitigation |
|---|---|
| `SequenceChatHandler.handleStepReturn` may need parameters not available in `_subscribeToScreenReturns` (e.g. profile snapshot, conversation context) | Read the handler signature first (T-53-02-01 prep step); thread missing params via constructor or Provider where needed. If signature needs widening, do it in the same plan with a follow-up commit. |
| Activating one sequence might cascade: e.g. `retirementPrep` step 2 routes to a screen that itself fires another return → infinite handler loop | Add a re-entrancy guard in `_subscribeToScreenReturns` (skip if a sequence-driven prompt was just injected within the last N ms). |
| Walker scenario doesn't cover `retirement_choice` intent | The walker scenario list (`sequence_template.dart:572-587` mappings) includes `retraite`; verify `walker.sh` types a prompt that maps to that intent. If missing, add a scenario string (small change in `walker.sh`). |
| Mocked profile in E2E test produces `Blocked` readiness on `/retraite` (the canonical screen for retirement_choice) | Seed the profile fixture with `age`, `canton`, `salaireBrut` per the readiness gate. Use the existing `_JULIEN_PROFILE` test fixture as a template. |
| `SequenceCoordinator.advance` may produce unexpected `SequenceAction` types not covered by the handler logic | Audit `sequence_coordinator.dart:19-67` (sealed `SequenceAction` class with 6 variants); ensure handler covers all 6 OR has a default-to-context-line fallback. |
