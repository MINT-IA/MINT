# Plan 53-03 — Tab 1 « Mes engagements & check-ins » surface (PLAN)

**Status:** Ready (independent of Plan 53-02; can ship in parallel)
**Companion CONTEXT:** `53-CONTEXT.md`
**Estimated effort:** 1–2 days

## Goal (one sentence)

Add one user-visible card to the Aujourd'hui tab that surfaces previously-silent persistent tools (commitments + monthly check-ins) so the user actually SEES that the chat's `record_check_in` / `show_commitment_card` actions persisted state — closing the « tool fires, persists, then disappears » UX gap Expert 5 documented.

## Why this plan now

Expert 5 (Engineering / Wiring Reviewer) flagged: « `apps/mobile/lib/screens/aujourdhui/aujourdhui_screen.dart:1-338` has zero references to `MonthlyCheckIn`, `CommitmentService`, `Earmark`, or `partnerEstimate`. Every `record_check_in` (`widget_renderer.dart:471`) and `show_commitment_card` (`widget_renderer.dart:652`) tool call persists silently and is then **invisible** outside the chat bubble that produced it. »

Plan 53-03 makes ONE of these surfaces visible. Earmarks + partnerEstimate are out of scope (separate plans).

## Tasks

### T-53-03-01 — New widget `commitments_and_checkins_card.dart`

**Files:**
- New `apps/mobile/lib/widgets/aujourdhui/commitments_and_checkins_card.dart`

**Behavior:**
- Stateless `Card` widget consuming `CoachProfileProvider.monthlyCheckIns` (existing field) + `CommitmentService.list()`.
- Renders up to 3 most-recent items per data source (configurable).
- Each item shows: icon (commitment vs check-in), one-line summary, relative timestamp (« il y a 2 jours »).
- Tap → `context.push('/coach/chat?session=<chatSessionId>')` resumes the conversation that produced the item (the `chatSessionId` is already persisted on each commitment / check-in record).
- Empty state: hide the card entirely (no « no data » placeholder — silent absence per Expert 5's risk note about empty-state spam).

**Visual treatment:** follow the existing `ContextualCard` pattern from Phase 6 (`MILESTONES.md` reference) — same border radius, padding, typography. **No new design panel needed** since this is reusing the existing card pattern; if any deviation is proposed, the 4-person design panel rule applies.

## Acceptance criteria (Plan 53-03 close-out)

1. New widget renders for profiles with at least one commitment OR check-in.
2. Tap on item navigates to `/coach/chat?session=<id>` with correct session.
3. Empty state hides card entirely (no spam).
4. ARB parity: 4 new keys × 6 locales = 24 entries; key counts match across locales (modulo `@meta` entries in fr).
5. `flutter analyze` clean. `flutter test` green on the new widget test.
6. WCAG AA on the new card surface (handled by existing `wcag_aa_all_touched_test.dart`).
7. HTML evidence updated.

## Risks + mitigations

| Risk | Mitigation |
|---|---|
| `CommitmentService.list()` may not exist or have a different name | Read `apps/mobile/lib/services/coach/commitment_service.dart` first; use whatever method returns `List<Commitment>`. If only persistence-side service exists, expose a read method (small additive change). |
| `MonthlyCheckIn` records may not carry a `chatSessionId` field | Check `apps/mobile/lib/models/monthly_check_in.dart`. If missing, the « tap → resume conversation » UX falls back to opening the coach with a generic resume intent; the data-model addition becomes a follow-up plan. |
| Tab 1 already crowded with 5+ cards (Phase 6 ContextualCard feed) | Position card AFTER the existing feed (deprioritize visually). User sees feed first, this card second — discoverable but not screen-stealing. |
| ARB parity drift across 6 locales (PR #439-class regression) | Pre-push checklist enforces `flutter gen-l10n` + key-count assertion (per `feedback_pre_push_checklist.md`). |
| Visual treatment too different from existing ContextualCard | Reuse the exact `ContextualCard` shell if architecturally feasible (compose, don't fork). If a new container is needed, design panel pre-push (per `feedback_design_panel_before_push.md`). |
