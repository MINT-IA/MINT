# Plan 53-01 — `ScreenRegistry × app.dart` parity audit + lint (PLAN)

**Status:** Ready
**Companion CONTEXT:** `53-CONTEXT.md`
**Estimated effort:** 1.5–2 days (lint + gap fill + coverage map)
**Dependency:** PR #443 (walker tooling) merged. PR #442 (Phase 52.3 truth-in-crypto) merged ✅.

## Goal (one sentence)

Close Handoff 2 Vague A by mirroring the existing `route_registry_parity.py` lint for the `MintScreenRegistry` (`apps/mobile/lib/services/navigation/screen_registry.dart`), filling the ~42-route gap between 153 `path:` declarations in `app.dart` and 111 `ScreenEntry` rows, and shipping a CI gate that prevents future drift.

## Why this plan first inside Phase 53

Plan 53-02 (Sequence wiring) will route through `RoutePlanner.plan(intentTag: ...)` which reads `MintScreenRegistry`. Routing into a screen that has no registry entry would silently fail — exactly the « façade-sans-câblage » class banned by Julien's discipline. 53-01 is the safety net for 53-02.

## Tasks

### T-53-01-01 — Build `tools/checks/screen_registry_parity.py`

**Pattern source:** `tools/checks/route_registry_parity.py` (337 lines, MAP-04, Phase 32-04). Mirror its structure, including:
- KNOWN-MISSES allow-list pattern (admin-conditional, nested children).
- Regex extraction of `path:` literals from `app.dart`.
- Comparison against `ScreenEntry(routePath: '...', ...)` literals from `screen_registry.dart`.
- `--extract-only` flag for manual auditing.
- Stdlib-only Python 3.9-compatible.

**Files:**
- New `tools/checks/screen_registry_parity.py` (~250 lines)
- New `tools/checks/screen_registry_parity-KNOWN-MISSES.md` (allow-list documentation)
- New `tests/checks/test_screen_registry_parity.py` (mirror `tests/checks/test_route_registry_parity.py`, fixtures-driven)

**Behavior:**
- Extract every `path:` literal from `apps/mobile/lib/app.dart` (already proven regex from `route_registry_parity.py:84`).
- Extract every `routePath:` (or whatever the `ScreenEntry` field is named — verify by reading `screen_registry.dart:50` `class ScreenEntry`) from `apps/mobile/lib/services/navigation/screen_registry.dart`.
- Symmetric diff: report routes-without-ScreenEntry AND ScreenEntries-without-route.
- Exit 0 on parity (modulo KNOWN-MISSES); exit 1 on drift; exit 2 on usage errors (sysexits.h EX_USAGE).
- Runtime budget: < 1 second (matches MAP-04 reference).

**Test:**
- `python3 tools/checks/screen_registry_parity.py --extract-only apps/mobile/lib/app.dart` → prints sorted list of 153 paths.
- Fixture-driven test: synthetic registry missing one route → exit 1 with `::error` diagnostic naming the missing route.
- Fixture-driven test: synthetic registry with extra ghost entry → exit 1 with diagnostic naming the ghost.

## Risks + mitigations

| Risk | Mitigation |
|---|---|
| Some `ScreenEntry` rows reference enums (`ScreenIntent`, `ScreenBehavior`) that may not have entries for all 42 missing routes | T-53-01-03 placeholder pattern with `// TODO 53-01: classify` + follow-up log; do NOT invent new enum values in this plan (separate plan). |
| The 42-route gap may include legitimately-orphan routes that should be DELETED, not registry-added | Document them in COVERAGE.md as « ORPHAN » class with a follow-up Phase 55+ deletion plan. Do NOT delete here — registry audit only. |
| `route_registry_parity-KNOWN-MISSES.md` patterns (admin-conditional, nested children) need re-applying for ScreenRegistry too | Mirror Category 7 (admin) + Category 5 (nested) from the route lint into the screen lint's KNOWN-MISSES doc. |
| Adding 42+ rows is mechanical → review fatigue → typos | Generate via a one-shot script if pattern is uniform, otherwise pair-review. Tests catch behavior regressions; lint catches typos. |
| Some routes use `path: _someFn(seg)` (Category 3 in route lint) | Same exemption pattern: structurally uncapturable, simply don't see them, document in KNOWN-MISSES. |
