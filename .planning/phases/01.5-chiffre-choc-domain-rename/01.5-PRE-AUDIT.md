## Audit 1 — Serialization (persisted state)

**Commands run:**
```
grep -rn '"chiffre_choc"' apps/mobile/lib/services/storage/   # dir does not exist
grep -rn '"chiffre_choc"' apps/mobile/lib/data/               # 0 hits
grep -rn '"chiffre_choc"' services/backend/app/db/            # dir does not exist
```

**Extended search (wire-contract level):**
```
grep -rn '"chiffre_choc"\|'\''chiffre_choc'\''' apps/mobile/lib services/backend
```

**Findings:**

1. `apps/mobile/lib/services/storage/` **does not exist**. MINT does not have a storage/ subdir — persistence is Hive/SharedPreferences scattered across providers, but none of them key on the literal `"chiffre_choc"` string.
2. `apps/mobile/lib/data/education_content.dart` has 0 string-literal hits (it uses `chiffreChoc` Dart identifiers, not JSON keys).
3. `services/backend/app/db/` **does not exist** — backend has no SQL layer (pure FastAPI + Pydantic + compute services). No DB columns, no migrations, no schema files to touch.
4. **Wire-contract layer (NOT persisted state, but matters):**
   - Backend dict payloads: 8 live hits in `services/backend/app/services/{unemployment/calculator, fiscal/church_tax_service, fiscal/cantonal_comparator, fiscal/wealth_tax_service, first_job/onboarding_service}.py` and `services/backend/app/api/v1/endpoints/{first_job, unemployment}.py`. These build dicts with `"chiffre_choc"` snake keys and pass them to Pydantic response models.
   - Pydantic schemas: **mixed convention** across `services/backend/app/schemas/`:
     - Some declare bare `chiffre_choc: str = Field(...)` (schemas/retirement, privacy, unemployment, arbitrage, fiscal, educational_content, family, commune). These rely on the backend-wide `alias_generator=to_camel` + `populate_by_name=True` (CLAUDE.md §4) to emit `chiffreChoc` on the wire.
     - Some declare explicit `chiffreChoc: ... = Field(..., alias="chiffreChoc")` (schemas/mortgage, pillar_3a_deep, debt_prevention, life_events). These emit `chiffreChoc` directly.
   - Mobile consumer: `apps/mobile/lib/services/api_service.dart:969` already reads `const ['chiffreChoc', 'chiffre_choc']` as a fallback list. This pattern proves the wire key was historically in flux and mobile was written defensively.
5. Mobile string-literal usage:
   - `apps/mobile/lib/screens/onboarding/chiffre_choc_screen.dart:185` — `screenName: 'chiffre_choc'` (analytics screen name string)
   - `apps/mobile/lib/services/coach/coach_orchestrator.dart:882` — `return 'chiffre_choc'` (routing key)
   - `apps/mobile/lib/services/coach/prompt_registry.dart:502` — `case 'chiffre_choc':` (routing switch)

**Gate decision:** **GO. Not a blocker.**

**Reasoning:**
- No persisted state (no SharedPreferences keys, no SQLite columns, no on-disk JSON) uses `"chiffre_choc"`. The default assumption in CONTEXT.md (pre-launch product, no persisted legacy) is confirmed. No read-time alias or one-shot migration is needed.
- The wire contract is **already `chiffreChoc` camelCase** for all endpoints via Pydantic's `to_camel` alias generator. The `chiffreChoc` JSON key on the wire is unchanged by renaming the internal snake-case `chiffre_choc` Python identifiers — because the alias is literal `"chiffreChoc"` where set, and `to_camel("premier_eclairage")` → `"premierEclairage"` where implicit.
- **HOWEVER:** the wire key itself IS changing (`chiffreChoc` → `premierEclairage` on JSON output), because the bare snake fields flow through `to_camel`, and the explicit-alias fields will be updated in B3. This is a breaking wire change. Per D-04 analytics decision ("hard rename, no dual-emit, pre-launch product"), this is aligned with the phase's posture: no mobile build in production relies on the old camel wire key.
- The mobile defensive read `['chiffreChoc', 'chiffre_choc']` will be **updated in F1 or F4** to `['premierEclairage', 'premier_eclairage']` (or just the camel key if the snake fallback was legacy cruft — confirm during F4).
- The three mobile string literals in onboarding screen / coach orchestrator / prompt registry are routing keys matching a backend routing identifier. They must be renamed in lockstep with the backend equivalent (`coach_models.py:18` — `chiffre_choc = "chiffre_choc"`). These three Dart string literals + the Python enum value = one coupled rename, assigned to commit B2 (backend enum) coordinated with F4 (mobile routing) — mobile compile-time cannot break between B2 and F4 because the routing key is a runtime dispatch string, not a type. **Add to the F4 commit checklist.**

**Scope note for 13-commit body:**
- B3 must re-run `generate_canonical.py` and diff to confirm `chiffreChoc` JSON keys become `premierEclairage` on the wire.
- F4 (services layer) must update `api_service.dart:969` fallback list to camel-only new key AND update the three coach routing string literals.
