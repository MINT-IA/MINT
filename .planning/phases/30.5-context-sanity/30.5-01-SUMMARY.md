## Decisions Made

- **Inline report.render() via lazy import** — dashboard.py imports render_report lazily inside each cmd_* handler. Keeps `--help` fast, prevents an upstream module bug from breaking unrelated subcommands.
- **CASE-WHEN over FILTER in metric SQL** — SQLite's `FILTER` clause requires 3.30+; Python 3.9 ships older SQLite on macOS. Rewrote the 4 metric queries to use `CASE WHEN ... THEN col END` which is functionally equivalent and portable.
- **Ingester failures are per-module, not fatal** — `cmd_ingest` catches exceptions per ingester and logs to stderr; the overall command still exits 0. Keeps the nightly cron idempotent even if one data source breaks temporarily.
- **Hit dedup via in-memory set, not SQL UNIQUE** — context_hits table schema didn't declare `UNIQUE(session_id, detected_at, hit_type, rule_id)`; ingest_hits.py loads existing keys at start and filters new rows. Cheaper than ALTER TABLE migration.
- **Baseline lockfile content is human-readable** — `captured_at: <iso>\ncommit_sha: <sha>` instead of JSON. Lets `cat .baseline-lock` be useful without tooling.

## Deviations from Plan

### Auto-fixed Issues

**1. [Rule 3 — Blocking] Stale SQLite journal held lock across ingest restart**
- **Found during:** Task 2 verification (first `dashboard.py ingest` run aborted with `database is locked`)
- **Issue:** A previous `dashboard.py ingest` subprocess from an earlier test run was still alive (PID 92400) and had an exclusive write transaction open. SQLite's journal file (`drift.db-journal`) persisted the lock visible to subsequent `sqlite3` CLI invocations.
- **Fix:** `pkill -f dashboard.py`, removed `.planning/agent-drift/drift.db*` stragglers, re-ran `init` + `ingest` cleanly. No code change required — the issue was background-task orchestration, not a dashboard defect. Added troubleshooting entry to `.planning/agent-drift/README.md` for future operators.
- **Files modified:** `.planning/agent-drift/README.md` (Troubleshooting section)
- **Committed in:** `a1ee5dbd` (Task 3 commit)

**2. [Rule 1 — Bug] `FILTER (WHERE ...)` not portable to SQLite <3.30**
- **Found during:** Task 3 first `dashboard.py report` invocation — Python 3.9 on macOS ships SQLite 3.28, which rejects the `FILTER` clause as a syntax error.
- **Issue:** 30.5-RESEARCH.md §Code Examples Example 2 uses `COUNT(DISTINCT c.sha) FILTER (WHERE ...)` per PostgreSQL/SQLite-3.30+ grammar. This crashes on stock macOS Python.
- **Fix:** Rewrote metric (a) and metric (b) queries to use `COUNT(DISTINCT CASE WHEN ... THEN col END)` — functionally equivalent, portable back to SQLite 3.7+. Documented in Decisions Made above.
- **Files modified:** `tools/agent-drift/render_report.py`
- **Committed in:** `8838d2df` (Task 1 commit — the fix landed at the same time as render_report was first created)
