# Phase 10.5 — Friction Pass 1 (Galaxy A14 device walkthrough)

> **"Très belle avant les humains" gate.** Julien runs the new S0 landing → intent chip → chat first message → first insight golden path on a real Galaxy A14 device. Every frottement is captured. Block-items reopen Phase 7/10 scope. Zero blockers before Phase 11.

---

## Goal (outcome, not task)

A Galaxy A14 owner (Julien, proxy for a real Swiss user with a mid-range Android phone) can go from app icon tap to first insight **without a single moment where they feel "I should already know this", "this is slow", "this is ugly", or "I don't know what to do next"**. Every remaining frottement is either fixed on the spot (block), queued for Phase 12 (polish), or deferred to a post-milestone backlog (nit).

## Why this phase exists

Phases 7 (Landing v2) and 10 (Onboarding v2) shipped code that looked right on a Flutter devtools preview and on a MacBook. But the target user is on a **Galaxy A14** — a 160 CHF mid-range Android phone with a modest GPU, a 90Hz LCD screen (not OLED), and real-world latency on Swiss mobile networks. Code that feels premium on a simulator can feel cheap, slow, or shameful on actual hardware. This phase is the forcing function that proves it.

The anti-shame doctrine is the north star: **any moment where the user might feel "I should already know this" or "I'm behind" is a BLOCKER, not a nit.** Polish is visual. Shame is existential. MINT cannot ship a single shame leak.

## Phase anatomy

```
Plan 10.5-01 (Claude, ~45 min) — Pre-walkthrough setup
  └─ Build A14 APK, install instructions, friction note template, recording setup

Plan 10.5-02 (Julien, ~30 min walkthrough + ~15 min note capture) — HUMAN-GATED
  └─ Cold start → S0 → intent → chat → first insight, note every frottement

Plan 10.5-03 (Claude, ~90 min) — Triage + hot-fix dispatch + re-verify
  └─ Tag block/polish/nit, dispatch blockers to Phase 7/10 code, Julien re-verifies
```

**Total estimated wall clock:** ~3-4 hours for a clean pass (zero blockers). Each blocker loop adds ~60-90 min (fix + re-install + re-verify).

## Decisions (LOCKED — plans must honor these)

### D-01 — Build profile: `release`, not `profile` or `debug`

**Decision:** The A14 walkthrough uses `flutter build apk --release --split-per-abi` (arm64-v8a APK).

**Rationale:**
- `debug` builds are 3-5× slower on mid-range Android → would poison the perf gate (cold start < 2.5s would fail on debug for reasons unrelated to actual product perf).
- `profile` builds keep observatory + some asserts → closer to release but still not what a user installs.
- `release` builds are what a real user gets from TestFlight/Play Store. This is the honest measurement.
- Tradeoff: no hot reload on device, no observatory for DevTools timeline. We accept this — stopwatch + self-report + video is enough for Phase 10.5. Full DevTools-based PERF baseline lives in Phase 12.

**Applies to:** Plan 10.5-01 Task 1 (build script), Plan 10.5-03 re-build after blocker fixes.

### D-02 — Recording method: Android native screen record + `adb pull`

**Decision:** Screen recording via Android 11+ built-in screen recorder (swipe-down tile), saved to `/sdcard/Movies/`, pulled via `adb pull` to `docs/FRICTION_PASS_1/walkthrough_1.mp4` (and `_2.mp4` for re-verify).

**Rationale:**
- Native recorder captures touches (if enabled in Developer Options → "Show taps"), is lossless (H.264), includes audio if Julien narrates, and is zero-friction to start/stop.
- No external camera, no reflection glare, no shaky hands.
- `adb pull` is trivial and leaves the recording on-device as backup.
- Fallback if `adb` is unavailable: AirDrop-equivalent (Google Drive upload from phone, download on Mac, move to repo). Claude will not commit the `.mp4` itself if it exceeds 100MB — instead, frame dumps via `ffmpeg` extraction every 0.5s to `docs/FRICTION_PASS_1/frames/`.

**Applies to:** Plan 10.5-01 Task 2 (install + record instructions), Plan 10.5-02 execution.

### D-03 — Friction note format: Markdown table (one row per frottement)

**Decision:** `docs/FRICTION_PASS_1.md` uses a single markdown table with columns:
`| # | Timestamp | Surface | Axis | Severity | Description | Proposed fix |`

**Rationale:**
- Easy to fill on a phone keyboard (one line per note, short fields).
- Diffable in git (YAML would collapse on structural changes).
- Claude can grep/parse it trivially for triage in Plan 10.5-03.
- One file = one walkthrough (Plan 10.5-02 produces `FRICTION_PASS_1.md`; the re-verify pass produces `FRICTION_PASS_1_REVERIFY.md`).

**Template lives in:** Plan 10.5-01 Task 3 (Claude creates the empty template with instructions at the top).

### D-04 — Block / Polish / Nit severity thresholds (PRECISE)

These are not fuzzy. Plans and triage MUST apply them literally.

| Severity | Definition | Examples | Disposition |
|----------|------------|----------|-------------|
| **block** | User CANNOT proceed, OR the user feels shame ("I should already know this" / "I'm behind" / "I'm stupid"), OR the app feels broken (crash, white screen, > 5s unresponsive), OR a core anti-shame / anti-jargon rule in `CLAUDE.md` is visibly violated, OR cold start ≥ 2.5s, OR first-frame-to-interactive ≥ 3s, OR first chat round-trip ≥ 4s. | Intent chip doesn't respond to tap; "LPP" shown without translation; loading spinner >3s without feedback; any word from the banned-terms list visible; text clips out of screen; contrast so low Julien squints. | Hot-fix immediately in Plan 10.5-03. Reopens Phase 7 or 10 scope. Re-verify on device. |
| **polish** | Noticeable but doesn't break flow. User proceeds. No shame leak. Quality bar issue — the "très belle" gate would fail but the "fonctionne" gate passes. | Animation timing feels 50ms too slow; card shadow too strong; microcopy is correct but flat; a tap target is 44dp instead of 48dp but still hittable. | Queue for Phase 12 pre-ship polish pass. Logged in `docs/POLISH_BACKLOG_P12.md`. |
| **nit** | Only Claude (or a designer with a loupe) would notice. A real user would never see it or care. | 1px misalignment on a card border; hex color off by one shade in a gradient stop; font-smoothing anti-aliasing difference between Android and iOS; a word that could be slightly more elegant. | Deferred to `docs/NIT_BACKLOG_POST_MILESTONE.md`. Do NOT fix in v2.2. |

**Shame test (mandatory mental check for every note):**
> "If a 55-year-old Swiss grandmother with no finance background saw this, would she think 'I'm stupid for not understanding' or 'I'm behind in life'?"
> If yes → **block**. Not polish. Not nit. **Block.**

### D-05 — Hot-fix dispatch protocol

**Decision:** Block-items reopen the relevant Phase 7 or Phase 10 scope via surgical commits on the CURRENT branch (`feature/v2.2-p0a-code-unblockers`), NOT new feature branches.

**Rationale:**
- We are in a long-running v2.2 milestone branch chain. Creating `hotfix/phase10.5-*` branches for every blocker fragments history.
- Each block-item becomes one commit: `fix(p10.5): <surface> — <one-line description>` referencing the friction note row number.
- If a blocker requires >1 file OR touches >1 surface, it gets its own short-lived branch `hotfix/p10.5-<slug>` merged back via squash.
- CI gates (flutter analyze, flutter test, pytest) must stay green between every hot-fix commit.

**Applies to:** Plan 10.5-03 Task 2.

### D-06 — "Until zero blockers" loop: max 3 iterations before escalation

**Decision:** The friction pass can iterate at most **3 times** (walkthrough → hot-fix → re-verify → hot-fix → re-verify → hot-fix → re-verify). If after 3 iterations there are still blockers, the phase escalates to Julien with a written "scope concerns" report and the option to:
(a) carve the remaining blockers into a new Phase 10.6,
(b) accept them as polish (explicit downgrade with written rationale), or
(c) extend to a 4th iteration with justification.

**Rationale:**
- Unbounded "until zero" loops can burn days and mask deeper design problems.
- If 3 hot-fix cycles don't converge, the issue is architectural, not cosmetic, and deserves a proper phase.
- 3 iterations × ~90 min per fix-and-reverify cycle = ~4.5 hours max before Julien sees an escalation report.

**Applies to:** Plan 10.5-03 Task 3 (re-verify loop control).

### D-07 — Performance gate measurement: stopwatch + `adb logcat` timestamp

**Decision:** Julien times with a physical stopwatch (phone stopwatch on a second device, or wall clock / wristwatch), cross-referenced with `adb logcat -v time | grep -iE "flutter|mint"` timestamps captured during the walkthrough. Three cold-start runs, median value wins. Flutter DevTools timeline is NOT required for this phase (deferred to Phase 12).

**Measurements:**
- **Cold start** = moment finger lifts off app icon → first pixel of S0 visible. Target < 2.5s.
- **First-frame-to-interactive** = first pixel visible → intent chips are tappable (tap registers and triggers state change). Target < 3s (cumulative from icon tap).
- **First chat round-trip** = tap "send" on first chat message → first character of assistant response streams in. Target < 4s.

If any metric fails → that's a **block** (per D-04).

**Applies to:** Plan 10.5-02 Task 2 (timed runs), Plan 10.5-03 triage.

### D-08 — Nit disposition: `docs/NIT_BACKLOG_POST_MILESTONE.md`

**Decision:** Nits go to a single append-only file, NOT into the issue tracker, NOT into roadmap phases, NOT into any v2.2 scope. They are out-of-milestone by definition. File is reviewed after v2.2 ships, at the start of v2.3 planning.

### D-09 — Golden path (exact sequence Julien tests)

The walkthrough is NOT open-ended exploration. It is this exact sequence, run 3 times cold (3 separate kill-app-and-cold-start runs):

1. **Cold start:** app fully killed (force-stop from Android settings), tap icon, stopwatch starts.
2. **S0 landing** appears. Julien reads it like a first-time user. Waits for any animation to complete.
3. **Intent chip tap:** picks the `explore` chip (default golden path). Stopwatch for tap→next-screen transition.
4. **Chat screen** appears. Julien reads the opener (coach's first message).
5. **First message send:** Julien types a canonical first message: *"Je viens d'avoir 30 ans, je commence à me demander si je devrais ouvrir un 3a."* Taps send. Stopwatch for send→first-character-of-response.
6. **First insight:** Julien reads the coach response in full. Any claim of certainty, any banned term, any untranslated jargon → block.

Surfaces NOT in this pass (deferred to later friction passes): ProfileDrawer, Explorer hubs, any life-event screen beyond the chat entry. Scope discipline.
