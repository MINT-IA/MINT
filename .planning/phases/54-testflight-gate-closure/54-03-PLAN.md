# Plan 54-03 — FAIL triage + ship to TestFlight staging (PLAN)

**Status:** Ready (depends on Plan 54-01 walker run producing FAIL triage tickets)
**Companion CONTEXT:** `54-CONTEXT.md`
**Estimated effort:** 2-3 days (variable — depends on Plan 54-01 FAIL count)

## Goal (one sentence)

Triage every FAIL produced by Plan 54-01's walker run, ship per-FAIL fixes (each as a focused PR), re-run the walker until all rows are PASS or explicitly SKIP-with-rationale, then bump `pubspec.yaml` to `2.9.0+N`, push to staging branch, watch `testflight.yml` succeed, confirm the new build processes in App Store Connect, and sign `docs/DEVICE_GATE_V27_CHECKLIST.md` GATE-01.

## Why this plan

Plan 54-01 produces evidence (PASS / FAIL / SKIP per row). Plan 54-03 closes the loop: turn FAILs into shipped fixes, prove TestFlight pipeline works end-to-end, sign the device gate that's been pending since Phase 7. Per `STAB-carryover.md:38` this is « BLOCKING TestFlight » — Plan 54-03 is the gate that unblocks it.

## Tasks

### T-54-03-01 — Triage every FAIL ticket

**Inputs:** `.planning/phases/54-testflight-gate-closure/triage/F-XX-*.md` (one per FAIL from Plan 54-01)

**Behavior per ticket:**
1. Root-cause analysis: read the surface file:line, check `git blame` for recent touches, classify failure mode:
   - **CRASH** (SimulatorCrash, FlutterError, AssertionError) — P0
   - **NAV-DEAD-END** (tap fires but screen doesn't change) — P0
   - **SILENT-NO-OP** (tap returns null / SizedBox.shrink) — P1
   - **VISUAL-REGRESSION** (renders but unexpected layout) — P2 (defer to Phase 55+ if not TestFlight-blocking)
   - **DATA-DEPENDENCY** (works only with seeded profile state) — SKIP with rationale
2. Decide: fix in this phase OR defer with explicit ACCEPT-WITH-RATIONALE entry in `triage/ACCEPT_LOG.md`.
3. Fix path: focused PR per surface, walker re-runs the row, ticket flips to PASS.

**Hard exclusions:**
- No new feature work — fixes only.
- No backend changes unless a FAIL traces to a backend bug.
- No refactor — minimal change to make the row PASS.

### T-54-03-02 — Re-run walker after each fix

**Behavior:**
- After each FAIL fix lands, run `bash tools/simulator/walker_audit_tap_render.sh --row <id>` to verify ONE row flips to PASS.
- Final pass before TestFlight push: `--all` run; results doc updated.
- Acceptance: zero FAILs OR every remaining FAIL has an `ACCEPT-WITH-RATIONALE` entry that the close-out audit accepts as non-TestFlight-blocking.

### T-54-03-03 — Bump version + ship to TestFlight staging

**Files:**
- `apps/mobile/pubspec.yaml` — version: `2.2.0+1` → `2.9.0+1` (per CONTEXT D-04)
- `apps/mobile/ios/Runner/Info.plist` — verify CFBundleShortVersionString matches (auto-derived from pubspec by Flutter; cross-check with `flutter build ios --no-codesign` output)
- `apps/mobile/ios/Runner/PrivacyInfo.xcprivacy` — verify all 12 declared data types still cover Phase 52.x + 53.x changes (Phase 53.x didn't add new flows, but Plan 54-02's chip wiring may add a new tap-event analytics class — audit this)

**Behavior:**
1. Bump pubspec.
2. Push to `staging` branch.
3. `testflight.yml` triggers automatically.
4. Watch CI: certificates load, Match cert sync, archive build, upload to App Store Connect.
5. Apple processes the build (~15-30min); confirm via `gh api repos/MINT-IA/MINT/actions/workflows/testflight.yml/runs --jq '.workflow_runs[0]'`.

## Risks + mitigations

| Risk | Mitigation |
|---|---|
| FAIL count exceeds 10 (Expert 5 estimate « expect 5-10 FAILs minimum »); triage takes longer than planned | Triage in priority order: CRASH > NAV-DEAD-END > SILENT-NO-OP > VISUAL-REGRESSION. Visual regressions can be ACCEPT-WITH-RATIONALE for TestFlight beta; users see them as bugs to file, not crashes. |
| Apple Connect rejects the build (missing entitlement, missing privacy declaration update from Phase 53.x) | testflight.yml dry-run on a sandbox profile first if practical. PrivacyInfo.xcprivacy already filled per Expert 5 evidence; Phase 53.x privacy work added new data flows that may need a manifest delta — audit in T-54-03-03. |
| Match cert sync fails (cert expired, signing identity drift) | testflight.yml has cert-sync error reporting wired; if it fails, file as P0 follow-up before signing GATE-01. Hold the version bump until certs are healthy. |
| `2.9.0+1` collides with an in-flight TestFlight build from a different branch | Verify no « 2.9.0+1 » exists in App Store Connect before push; bump to `2.9.0+2` if collision. |
| Walker re-runs are slow (each ~30min) → triage iteration is painful | Use `--row <id>` mode to exercise one row at a time after a single-row fix. Full `--all` run only as the final ship gate. |
