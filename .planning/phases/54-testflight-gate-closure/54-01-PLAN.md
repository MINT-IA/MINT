## Goal (one sentence)

Replace the « 100% TODO — manual » walker scaffold (`AUDIT_TAP_RENDER.md`, scaffold authored 2026-04-07) with a real autonomous walker run on dev tip + iOS sim that traverses every primary-depth interactive element in 3 tabs + drawer (~56 rows), captures screenshot evidence per row, asserts non-blank render via sha256 distinctness + Sentry breadcrumb, and writes back PASS/FAIL/SKIP verdicts to a generated results doc.

## Tasks

### T-54-01-01 — Build `tools/simulator/walker_audit_tap_render.sh`

**Files:**
- New `tools/simulator/walker_audit_tap_render.sh` — Bash, mirrors the architecture of `tools/simulator/walker.sh` (cliclick + idb helpers, `--dry-run` mode, `--archetype <slug>` for seed-driven runs).
- New `tools/simulator/audit_tap_render_rows.tsv` — committed TSV of the 56 rows, columns: `id` (e.g. 1.1), `surface_file_line`, `expected_text`, `tap_strategy` (centered_tap / scroll_tap / text_match), `tap_x`, `tap_y`, `wait_quiescent_max_iters`, `verify_breadcrumb` (optional Sentry breadcrumb name to match).
- New `tools/simulator/test_walker_audit_tap_render.sh` — pure-shell smoke test asserting (a) row TSV parses cleanly, (b) `--help` lists the 3 tab + drawer sections, (c) `--dry-run` produces 56 « would tap » lines without booting sim.

**Behavior:**
- `bash tools/simulator/walker_audit_tap_render.sh --archetype swiss_native --tab 1` runs all Tab 1 rows.
- `... --row 2.3` runs a single row (debug mode).
- For each row: build sim with `--dart-define=MINT_E2E_ARCHETYPE=...` + `MINT_WALKTHROUGH_PHASE=54`, install, launch, navigate to the row's tab, perform the tap, wait for UI quiescence (up to N polls), screenshot to `screenshots/audit-tap-render/<archetype>/<id>-<surface>.png`, compare sha256 against pre-tap screenshot to assert non-blank render.
- PASS = sha256 changed AND no error breadcrumb in Sentry within 2s window.
- FAIL = sha256 unchanged OR FlutterError / SimulatorCrash / AssertionError in breadcrumbs.
- SKIP = explicit row metadata `skip_reason: <text>` (e.g. « opens external URL », « device-only feature »).

### T-54-01-04 — Wire walker into CI as scheduled run (not blocking)

**Files:** new `.github/workflows/walker_nightly.yml`

**Behavior:**
- Scheduled cron `0 3 * * *` (3am UTC, off-peak).
- Runs on `macos-latest` (sim availability).
- Boots iPhone 17 Pro sim, installs build from latest dev, runs `walker_audit_tap_render.sh --archetype swiss_native --all`.
- Uploads `AUDIT_TAP_RENDER_RESULTS.md` + screenshots as artifacts.
- Posts result summary (PASS/FAIL/SKIP counts) to a GitHub issue.
- **Non-blocking** for PR merges (advisory until Plan 54-03 lands the FAIL triage).

**Hard exclusions:**
- Walker is read-only on the app — no state mutation, no SharedPreferences writes, no backend calls.
- Android sim is OUT OF SCOPE (per CONTEXT D-04 — Phase 55+).
- Pixel-perfect golden comparison is OUT OF SCOPE — sha256 distinctness is the gate.
- Performance / cold-start budget tuning is OUT OF SCOPE.

## Risks + mitigations

| Risk | Mitigation |
|---|---|
| iOS sim build fails because of code-signing / xattr regressions (recurring per `ENV_WORKTREE_PROVENANCE.md`) | Walker uses `--no-codesign` flag from Plan 53-01's walker.sh pattern; provenance xattr handling already proven. |
| Tap coordinates drift across Flutter version upgrades | Periodically re-calibrate by capturing a clean « landing » screenshot and updating TSV `tap_x`/`tap_y` columns. Phase 55+ task. |
| Some rows are inherently flaky (network-dependent screens, animation-tied taps) | Mark as `SKIP` with explicit reason in TSV; close-out gate is « walker runs without crashing », not « 100% PASS ». |
| Walker overhead per row is too high → 1+ hour per run | Parallelize across 3 sims (3 archetype seeds in parallel); each sim handles its tab subset. Defer to optimization phase if first run fits in 30min. |
| Sentry breadcrumb match requires backend connectivity (Railway staging) | Walker hits Railway staging per `feedback_app_targets_staging_always.md`; staging is the canonical target. If staging is down at run time, walker FAILs all rows that depend on backend (nav-only rows still PASS). |
| LLM dependency: Coach tab rows that emit `route_to_screen` chips depend on the LLM responding deterministically | Use stubbed coach response fixtures via `MINT_E2E_LLM_FIXTURE` env var; only the coach-input-bar rows actually hit the LLM. Most chip rows can use pre-canned tool calls injected via dart-define. |
