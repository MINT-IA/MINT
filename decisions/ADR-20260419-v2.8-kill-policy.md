# ADR: v2.8 Kill-Policy — No Stabilisation-Sequel

**Date:** 2026-04-19
**Status:** Accepted
**Milestone:** v2.8 L'Oracle & La Boucle
**Supersedes:** — (first explicit kill-policy for MINT)

---

## Context

MINT shipped:
- v2.4 "Fondation" (2026-04-12) — labelled "infrastructure fix"
- v2.6 "Le Coach Qui Marche" (2026-04-13) — labelled "coach works"
- v2.7 "Coach Stabilisation + Document Digestion" (2026-04-14, awaiting device gate) — labelled "stabilisation"

Each time, the milestone was framed as "finishing" — and each time a successor milestone was created to finish what the last one didn't finish. The label changed, the work pattern did not. Dead features and façade-without-wiring accumulated: Wave E-PRIME had to remove **42K LOC** of dead code in 2026-04-18 to recover.

v2.8 "L'Oracle & La Boucle" is the next in line. Without an explicit commitment, it will default to the same pattern: a "v2.9 Stabilisation" will silently materialise to finish v2.8's gaps.

## Decision

**If v2.8 exits with any table-stake requirement unmet, we KILL the feature via flag. We do NOT create a v2.9 "stabilisation" to finish it.**

Concretely:
1. Every Phase 36 P0 requirement (`FIX-01` through `FIX-09`) has a **kill-switch flag** provisioned in Phase 33 BEFORE Phase 36 begins.
2. On v2.8 close, if the requirement is not verifiably met (green lint + device-walkthrough signed by Julien + regression test landed), the corresponding flag is set `false` in production.
3. The flag stays off until the feature is revived in a future **feature milestone** (not a stabilisation milestone). Reviving requires a new ADR justifying why it's worth the cost.
4. A "stabilisation milestone" is no longer a valid milestone type in MINT's roadmap. Stabilisation becomes a **transversal discipline** (compression at every phase), not a phase or milestone of its own.

## Budget floor (non-borrowable)

To prevent the déjà-vu pattern mechanically:

| Phase | Min budget | Borrowable? |
|-------|-----------|-------------|
| 31 Instrumenter | 1.5 sem | from 34 only |
| 34 Guardrails | 1.5 sem | from 31 only |
| 32 Cartographier | 1 sem | from 33 only |
| 33 Kill-switches | 1 sem | from 32 only |
| 35 Boucle Daily | 1 sem | — |
| **36 Finissage E2E** | **2 sem MINIMUM** | **never — non-borrowable** |

If Phase 31-35 collectively overrun beyond their combined budget, Phase 36 does not absorb the overrun. Instead, **we descope differentiators** (listed in FEATURES.md §J) in the documented order.

## Consequences

**Positive:**
- Forces honest sizing of phases 31-35 (no magic "we'll finish in 36")
- Users never see broken features — flag off is better than "mostly works"
- Compression discipline becomes cultural (compression = doctrine, not milestone)
- v2.9 scope is free — it can be feature work (the deferred "La Confiance" idea, Privacy Nutrition Label, etc.), not stabilisation debt

**Negative / accepted costs:**
- Some features killed in v2.8 may never revive — that's explicitly okay. A killed feature is better than dead-code accumulation.
- No escape hatch: if Phase 36 genuinely needs more time, we ship fewer phases, not a successor milestone.
- Requires discipline to say "no" when user (= creator = Julien) wants to add "one small thing"

## Enforcement

- This ADR is referenced in `PROJECT.md` Key Decisions table
- Every Phase 36 REQ in REQUIREMENTS.md has a `kill_flag:` field naming its kill-switch
- Closing the milestone requires Julien to sign: `v2.8-close: killed=[flag1, flag2, ...] shipped=[req_id1, req_id2, ...]` in a commit message. The set must be exhaustive (every non-shipped REQ has a kill entry).
- Violations of this policy (creating a "v2.9 stabilisation" milestone without ADR superseding this one) are caught by `tools/checks/milestone_discipline.py` — a new lefthook gate added in Phase 34.
